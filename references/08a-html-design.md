# Step 8a: HTML Landing Page

Generate a **pixel-perfect, beautifully designed landing page** — a single vertical HTML document that presents the full narrative as a web experience. This page is a **standalone deliverable** in its own right and serves as the **design reference** for PowerPoint conversion.

This is NOT a slide deck displayed vertically. It is a landing page inspired by the best examples of modern web design — long-scroll storytelling, bold typography, striking visual composition, seamless content flow. Think Stripe, Linear, or a top agency's case study page.

## Prerequisites

Before reading this file, Gates 1-2 must be complete:
- Gate 1: Storyline approved (headline sequence locked)
- Gate 2: Sceptical review passed (all MUST FIX items resolved)
- Content distillation complete (per `08-slides-content.md`)

## Gate 3: Collect Visual Style + Brand

Use **AskUserQuestion** to collect visual style and brand preferences right before HTML generation.

### Q1 — Visual Style (header: "Visual Style")

Options: Clean & Minimal / Data-Heavy / Visually Rich / Corporate/Branded (custom).

Style definitions:
- **Clean & Minimal**: 40%+ white space, one accent color, minimal gridlines, no decoration. Elegant restraint.
- **Data-Heavy**: Dense charts/tables, multiple visualizations per section, muted palette. Information-rich.
- **Visually Rich**: Framework diagrams, process flows, icons, color-coded categories. Expressive and layered.
- **Corporate/Branded**: Apply user's brand colors, fonts, logo, match existing materials.

### Q2 — Brand Styling (header: "Brand")

Check `references/styles/` for available brand profiles. Options: each found brand + "None" + "Custom."

- If a brand is selected, read its file and apply all specified properties.
- If "Custom," collect brand elements from the user and create a new file from `references/brand-styling-template.md`.
- If "None," use the design direction from Q1 with a bold, content-informed palette.

### Gate 3 Checkpoint

Before generating HTML, confirm the visual style and brand choices back to the user. Summarize: chosen style, brand (if any), colors/fonts that will be used. Get explicit approval. Do not proceed without this.

## Design Philosophy

### This Is a Landing Page, Not a Deck Preview

The old approach: 16:9 cards stacked vertically, each mimicking a PowerPoint slide. **Do not do this.**

The new approach: a single, continuous, beautifully designed web page where content flows naturally through full-width sections, hero areas, sticky elements, dramatic reveals, and varied layouts. The content matches the approved storyline, but the presentation is native to the web.

The PowerPoint conversion (Gate 5) will **interpret** this page's design language — colors, typography, spatial hierarchy, visual treatments — and translate it into slides. The HTML does not need to look like slides. It needs to look like the best landing page you've ever seen.

### Design Benchmarks

Before coding, study the visual language of:
- **Stripe**: Clean data presentation, subtle gradients, systematic typography
- **Linear**: Dark-mode sophistication, sharp hierarchy, purposeful motion
- **Apple product pages**: Hero sections, bold stats, immersive storytelling
- **Top consulting/agency case studies**: Authority, clarity, elegant data viz

The page should feel like it was designed by a studio, not generated by an AI.

## Page Structure

Generate a **single self-contained HTML file** with inline CSS and JS. The page flows as a continuous narrative — no cards, no boxes, no simulated slides.

### Section Architecture

Each section of the page corresponds to a point in the approved storyline. But sections are NOT uniform containers — they vary in layout, density, and visual treatment based on the content.

```
┌──────────────────────────────────────────────────┐
│  HERO                                            │
│  Title, subtitle, a single commanding visual     │
│  Full viewport height, immersive                 │
│                                                  │
├──────────────────────────────────────────────────┤
│  EXECUTIVE SUMMARY (if selected)                 │
│  Key message + supporting points                 │
│  Tight, high-density, above-the-fold feel        │
│                                                  │
├──────────────────────────────────────────────────┤
│  CONTEXT (if selected)                           │
│  Why we're here, what was asked                  │
│                                                  │
├──────────────────────────────────────────────────┤
│                                                  │
│  CONTENT SECTIONS                                │
│  Each section: full-width, varied layout         │
│  - Argument sections: headline + evidence        │
│  - Data sections: charts, stat callouts, tables  │
│  - Comparison sections: side-by-side layouts     │
│  - Roadmap sections: timeline visualizations     │
│  - Framework sections: diagrams, architecture    │
│                                                  │
│  Sections alternate visual treatments:           │
│  light/dark backgrounds, accent bands,           │
│  full-bleed areas, contained prose blocks        │
│                                                  │
├──────────────────────────────────────────────────┤
│  CLOSING / NEXT STEPS                            │
│  Strong close with clear call to action          │
│                                                  │
├──────────────────────────────────────────────────┤
│  APPENDIX (if selected)                          │
│  Detailed supporting material                    │
│                                                  │
└──────────────────────────────────────────────────┘
```

### Section Types and Treatments

**Hero section**: Full viewport height. Title in commanding typography. Subtitle or key message. A single strong visual — gradient, abstract shape, or brand-aligned background. No clutter.

**Stat callout sections**: 2-4 large numbers with labels, arranged in a grid or row. Numbers should be visually dominant — 4-6x the body text size. Use these to punctuate the narrative between argument sections.

**Argument sections**: Headline (action title) + structured content (bullets, short text, paired visual). These map most directly to argument slides in the deck.

**Comparison / evaluation sections**: Side-by-side layouts, tables, or matrix grids. Web-native: can use hover states, highlighted rows, or toggle views.

**Roadmap / timeline sections**: Horizontal or vertical timeline visualizations. Swimlanes, progress bars, phased blocks. Web-native: can scroll horizontally or reveal phases on scroll.

**Framework / architecture sections**: Diagrams, layered visuals, icon grids. Full-width or contained within a focused area. Web-native: can be interactive or animated.

**Transition / divider sections**: Short, full-width bands with a single statement or quote. Used to shift the narrative mood or introduce a new argument thread. Dark background, centered text, dramatic.

### Content Mapping

Each section must have a `data-section` attribute with a sequential number (1-indexed) and a `data-content` attribute describing its role. These attributes allow the PowerPoint conversion to identify content boundaries and types.

```html
<section data-section="1" data-content="hero">...</section>
<section data-section="2" data-content="exec-summary">...</section>
<section data-section="3" data-content="argument">...</section>
<section data-section="4" data-content="stats">...</section>
<section data-section="5" data-content="comparison">...</section>
<section data-section="6" data-content="roadmap">...</section>
<section data-section="7" data-content="transition">...</section>
<section data-section="8" data-content="framework">...</section>
<section data-section="9" data-content="closing">...</section>
<section data-section="10" data-content="appendix">...</section>
```

**Important**: The number of HTML sections will NOT equal the number of PowerPoint slides. A single argument section on the page might become one slide. A dense comparison section might become two slides. A stat callout section that punctuates the web narrative might be merged into an adjacent slide. The `08b-pptx-convert.md` handles this mapping — the HTML just needs clear semantic structure.

## Semantic HTML for Conversion

Use semantic markup so the PowerPoint converter can extract content:

| HTML Element | Purpose | Maps to in pptx |
|---|---|---|
| `<h1>` | Section headline (action title) | Slide title |
| `<h2>` | Sub-section header | Section header on slide |
| `<p>` | Body text | Body text |
| `<ul>`, `<ol>` | Structured lists | Bullet/numbered lists |
| `<blockquote>` | Key message / pull quote | Callout / key message |
| `<figure>` + `<figcaption>` | Chart/visual with caption | Chart/image with caption |
| `<table>` | Tabular data | Table |
| `<footer>` inside section | Source citation | Source citation line |
| `<div class="stat-callout">` | Large number + label | Large stat callout |
| `<div class="columns">` | Multi-column layout | Side-by-side content |
| `<div class="icon-row">` | Icons with labels | Icon + text row |

## Frontend Aesthetics Guidelines

### Typography

- Choose fonts that are distinctive and elevate the design. Avoid generic defaults (Arial, Inter, Roboto) unless specified by a brand file.
- Pair a display/heading font with a clean body font.
- Use Google Fonts or system fonts that have wide availability.
- Extreme size contrast: hero titles should be 3-5x body text. Section titles should feel commanding. Use font weight and letter-spacing to create hierarchy beyond just size.
- Line height: generous for body text (1.6-1.8), tight for large headlines (1.0-1.2).

### Color and Theme

- Commit to a cohesive palette. Use CSS variables for consistency.
- Dominant color with sharp accents outperforms timid, evenly-distributed palettes.
- **Alternate section backgrounds**: Light and dark sections should alternate or be grouped to create visual rhythm. Not every section should have the same background.
- Use color to signal content type: dark for narrative/transitions, light for data/evidence, accent for key messages.

### Spatial Composition

- **Full-bleed sections**: Some sections span the full viewport width with background color/image. Content within uses `max-width` for readability.
- **Contained sections**: Others sit within a narrower content column for focused reading.
- **Generous vertical padding**: Sections need breathing room — 4-8rem between them. Don't cram.
- **Asymmetric layouts**: Two-column sections where one column is 60/40 or 70/30, not always 50/50.
- **Grid-breaking moments**: A full-width stat callout interrupting a contained narrative. A pull quote that bleeds into the margin. A timeline that scrolls horizontally within a vertical page.

### Backgrounds and Visual Details

- Create atmosphere: gradient meshes, subtle textures, geometric patterns, layered transparencies.
- Each section should feel like it belongs on the same page but has its own character.
- Use borders, accent lines, and shapes as structural elements, not decoration.

### Motion (Optional)

- Subtle scroll-triggered reveals: fade-in, slide-up for sections and key elements.
- Counters for stat callouts (numbers counting up as they scroll into view).
- Keep it restrained — this is a consulting deliverable, not a marketing site. But motion done well adds polish.

### What to Avoid

- **Generic AI aesthetics**: Overused fonts, purple gradients on white, cookie-cutter layouts.
- **Uniform sections**: If every section has the same width, padding, background, and layout, the page feels templated. Vary treatments.
- **16:9 cards or boxes**: The page is not a deck preview. No card containers, no aspect-ratio constraints, no simulated slides.
- **Centered body text**: Left-align paragraphs and lists. Center only hero titles, stat callouts, and transition statements.
- **Low-contrast text or elements**.
- **Accent lines under titles** (hallmark of AI-generated design).
- **Wall of text**: Even on a web page, the slide content rules from `08-slides-content.md` apply to structured content. Bullets are short. Tables are scannable. Prose is minimal.

## Gate 4: HTML QA

After generating the HTML file, run automated QA before presenting to the user.

### Automated Visual QA

Use the webapp-testing skill to:

1. Open the HTML file in a browser
2. Take a full-page screenshot
3. Take screenshots at key section positions (hero, exec summary, each major content section, closing)

Launch a **QA subagent** to inspect the screenshots with this prompt:

```
Visually inspect this HTML landing page. It presents a consulting deliverable as a beautifully designed web experience. Assume there are issues — find them.

This should look like a pixel-perfect, studio-designed landing page — not a slide deck displayed vertically, not an AI-generated template. Judge it against the standard of the best designed web pages (Stripe, Linear, Apple product pages, top agency case studies).

Look for:
- Sections that look templated or repetitive (same layout, same background, same spacing throughout)
- Typography issues: weak hierarchy, inconsistent sizing, generic font choices
- Color issues: low contrast, muddy palette, too many competing colors
- Spatial issues: cramped sections, insufficient padding, elements too close together
- Content overflow, text cut-off, or overlapping elements
- Sections that feel visually empty or unfinished
- Sections that feel like text dumps without visual structure
- Inconsistent alignment between sections
- Brand color/font mismatches (if a brand was selected)
- Mobile-unfriendly layouts (though desktop is primary, basic responsiveness expected)
- Anything that looks AI-generated rather than designer-crafted: accent lines under every title, identical section layouts, purple-on-white gradients, generic icon grids

For each section, list issues found. Be specific about which section and what the problem is. Also note what's working well.
```

### Fix and Re-verify

1. Fix all issues found
2. Re-screenshot affected sections
3. Re-inspect until a full pass reveals no new issues

### Present to User

After QA passes, present the HTML file to the user:
- Note that this is both a standalone artifact and the design reference for the PowerPoint
- Ask if they want any visual changes before proceeding to PowerPoint conversion
- Get explicit approval before moving to Gate 5
